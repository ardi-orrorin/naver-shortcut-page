## 버전 관리

1. release/v<버전> 로 배포
2. 버전은 1.0.0 으로 시작

## 코드 수정

0. 코드 수정 후에는 항상 `npm run lint`를 실행해 확인한다.
1. 새 브랜치 이름 규칙
   - 기본: `<현재 브랜치>/fix-<수정할 내용>`
   - 현재 브랜치가 `main`일 때: `dev/fix-<수정할 내용>/main`
   - 이슈 번호 기준: `<현재 브랜치>/fix-<issue 번호>`
2. 브랜치 병합은 개발자가 승인 후 직접 진행한다.

## 작업 전 체크리스트

1. `git branch --show-current`로 현재 브랜치 확인
2. `main` 또는 규칙과 다르면 즉시 새 브랜치를 생성한 뒤 작업 시작
3. 브랜치 분기 없이 코드를 수정하지 않는다

## 코드 수정 주의사항

1. 인 코드 최대한 재활용
2. 코드 최대한 간결하게 작성
3. 코드 추가시 다른 쪽에서 재활용 할 수 있도록 확장성 있는 코드로 작성
4. 유지보수 관점에서 코드를 작성한다.


## 브렌치 병합 요청처리 방법

1. 병합 요청 시 항상 상위 브랜치로 병합한다. 메시지에는 요청한 수정 내용을 명시한다.
2. 예: `dev/fix-<issue 번호>/fix-<수정 내용>`처럼 하위의 하위 브랜치라면 한 단계 상위 브랜치로 병합한다.
